Payload-Gen Project Summary
===========================
Schema-grounded payload generator for asset-finance APIs.
The API spec is always an input at inference time — never memorized in model weights.

Phase 0 — Project setup
-----------------------
Achieved: Created folder skeleton, requirements.txt, README stub, git repo.
Decision: MLX stack on Apple Silicon (M5 Air, 16GB).
Result: Environment install confirmed working.

Phase 1 — Smoke-test base 3B model
----------------------------------
Achieved: Ran mlx-community/Llama-3.2-3B-Instruct-4bit locally.
Observation: ~1.9GB peak memory; generation ~1s after initial download.
Result: Base model runs; suitable for QLoRA fine-tuning.

Phase 2 — Data generator (Layer C foundation)
---------------------------------------------
Achieved: Built invent_spec, pick_task, render_request, emit_target, build_dataset.
Generated: 6,000 train + 800 valid JSONL rows (MLX chat format).
Decision: Removed post_payment endpoint (redundant with post_receipt).
Decision: ~25% distractor rate, ~12% impossible/refusal rows in generator.
Observation: Every row uses a unique random spec (different seeds) — model cannot memorize APIs.
Result: Unlimited mechanically-verifiable training data.

Phase 3 — Baseline (before training)
------------------------------------
Achieved: Generated test.jsonl (800 rows, seed 2_000_000). Built score.py + baseline.py.
Result (base model, unseen test specs):
  - Schema-valid: 0.0%
  - Exact match: 0.0%
  - Refusal accuracy: 0.0%
Observation: Base model outputs body-only JSON, markdown fences, hallucinated endpoints.
Decision: Baseline captured before training to make later gains credible.

Phase 4 — LoRA fine-tuning
--------------------------
Achieved: 600 iterations, batch-size 1, 8 LoRA layers, lr 2e-4, grad-checkpoint.
Training: Val loss 2.681 → 0.281; flattened ~iter 300.
Decision: Built capture_examples.py to snapshot same 2 rows at baseline / iter-300 / final.
Observation (examples.json):
  - Baseline: wrong format, wrong refusals
  - Iter 300: perfect valid payload; refusal still wrong
  - Final: correct refusal; valid payload schema-valid but distractor field leaked
Result: Model learned output format + refusals; distractors remain weak spot.

Phase 5 — Fine-tuned demos on unseen specs
------------------------------------------
Achieved: phase5_demo.py — 3 cases from test set (seeds 2M+).
Results:
  (a) Valid post_receipt: schema-valid yes, exact match no (value/date parsing)
  (b) Distractor book_lease: schema-valid yes, exact match no (in_service_date leaked)
  (c) Impossible transfer_ownership: exact refusal — perfect
Observation: Skill is spec-reading, not memory — test specs never appeared in training.

Phase 6 — Eval harness + benchmark
----------------------------------
Achieved: Extended score.py with distractor-specific metrics (DistExact%).
Achieved: scripts/benchmark.py — base vs fine-tuned; seen (valid) vs unseen (test) splits.
Achieved: mock_server.py — FastAPI validates payloads via jsonschema.
Result (800 unseen test rows, 100 seen valid rows):
  base-3B:        0% schema / 0% exact / 0% refusal
  fine-tuned-3B: 95% schema / 47% exact / 99% refusal (unseen)
  Distractor gap: 13 points (non-distractor exact 50% vs distractor exact 37%)
Decision: Gap below 15-point threshold — extra distractor training optional, not urgent.
Note: Claude skipped (no ANTHROPIC_API_KEY); set key and rerun for frontier comparison.
Observation: Fine-tuning delivered huge validity gain; exact match limited by NL→value parsing.
Result: Mechanically verifiable eval — no human judgment in scoring.

Phase 7 — Layer A: retrieval
----------------------------
Achieved: 12 fictional specs in specs/ (seeds 500001–500012, unique filenames).
Achieved: src/retrieval/retrieve.py — embed with all-MiniLM-L6-v2, cosine match on request.
Decision: Spec library is fetched at runtime, not fine-tuned — spec set can change without retraining.
Observation: Whole-spec embedding confused similar APIs (book lease → post receipt).
Improvement: Endpoint-level retrieval (one vector per operation) + keyword boost (+0.30).
  - "book a lease" → book_lease (was matching post_receipt on whole-spec embed)
  - "post receipt" → post_receipt; "amend contract" → amend_contract
Result: Retrieval picks spec AND operation before Layer C generates the payload.

Phase 8 — Layer B: Excel → fields
---------------------------------
Achieved: src/mapping/excel_map.py — fuzzy column match, type coercion (dates, currency).
Achieved: data/sample_deal.xlsx — messy bank-style headers (Contract #, Eqpt Code, etc.).
Decision: Mapping is deterministic code, not fine-tuned — rules + aliases, not ML.
Observation: Fixes exact-match pain by supplying clean values instead of parsing NL in the model.
Result: Spreadsheet row → cleaned values dict ready to merge into Layer C payload body.

Phase 9 — Orchestration + chat wiring
-------------------------------------
Achieved: src/orchestration/decompose.py — rule-based multi-step (e.g. "book a deal" → create_asset + book_lease).
Achieved: src/app/server.py — FastAPI full pipeline (retrieve → map Excel → decompose → Layer C).
Achieved: src/app/chat.html — minimal browser UI; scripts/start_demo.sh starts model + app.
Flow: request (+ optional Excel) → Layer A spec/operation → Layer B values → orchestration ops
      → fine-tuned model @ :8080 per operation → JSON payloads returned.
Decision: Layer C served separately via mlx_lm.server; app calls OpenAI-compatible /v1/chat/completions.
Result: End-to-end demo at http://localhost:8000 with four layers wired.

Governing idea (unchanged)
--------------------------

Retrieval fetches specs; mapping handles Excel; only Layer C is fine-tuned.
